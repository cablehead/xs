## Highlights

This release makes actors process frames roughly twice as fast and fixes a slow memory leak in their hot loop.

Each actor now runs its event loop on one dedicated OS thread, evaluating its closure inline. Previously every frame bounced from an async task to an engine worker thread and back over a oneshot channel; that round trip, two context switches per frame, was the dominant per-frame cost. On the new bench, an actor threading state through a trivial closure went from ~40K to ~90K frames/s (25.2us to 11.2us per frame). Replays of large backlogs, such as rebuilding projections with `start: "first"`, see the same per-frame saving.

The leak: every closure evaluation registered a job in the engine's jobs table and never removed it, so a long-running actor's table grew by one entry per frame processed. The actor loop now attaches one long-lived job for its lifetime. A regression test pins this.

Topic filtering on reads now accepts multiple patterns, comma separated, as one capability shared by every surface: `.cat -T "game.move.*,game.create"` (or a nushell list, `-T [game.move.* game.create]`), `xs cat --topic`, the HTTP `topic` query param, and a new optional `topics` field in actor configs. A single pattern keeps the existing indexed fast path; multiple patterns union the per-pattern indexed scans, merged in id order. Actors with a `topics` filter skip frames they do not care about at the read level, before closure eval: on the bench's mixed stream (10% relevant), a filtered actor processes the stream at ~2.4us per stream frame versus ~9.3us unfiltered. Lifecycle frames and the synthetic `xs.threshold` / `xs.pulse` frames always reach a filtered actor. Commas are not valid in topic names, so the list form is unambiguous.

xs also gains a benchmark suite: `cargo bench --bench throughput` measures four dimensions over 100K frames, each printed as one parseable line:

- `append`: raw store writes
- `replay`: historical scan (`read_sync`)
- `actor-state`: per-frame actor framework overhead
- `actor-emit`: the buffered `.append` output path

Numbers are only comparable on the same hardware, but the format is stable, so release-over-release tracking on a fixed box is just "run and append to a log".

The actor refactor, measured with this bench*:

| dimension   | 0.13.3            | 0.13.4            | change    |
|-------------|-------------------|-------------------|-----------|
| append      | 51.1K/s (19.6us)  | 50.0K/s (20.0us)  | unchanged |
| replay      | 1.25M/s (0.80us)  | 1.85M/s (0.54us)  | +48%      |
| actor-state | 39.7K/s (25.2us)  | 89.5K/s (11.2us)  | 2.26x     |
| actor-emit  | 6.2K/s (160.9us)  | 7.2K/s (139.9us)  | +16%      |

\* `cargo bench --bench throughput` on the same idle 12-core Linux box, runs sequential not concurrent, 100K frames per dimension, per-frame cost in parentheses. The 0.13.3 column is the same bench file run against the pre-refactor tree.

This release is non-breaking; upgrading from 0.13.3 requires nothing.
